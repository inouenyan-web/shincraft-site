### 🔴 全員検討の原則（2026-06-08 追加・ピラミッド式委譲の禁止）

判断が必要なタスク（手段選び・委譲・井上さんへの作業依頼を伴うもの）では、
**1セッションの単独判断で直列に委譲してはならない**（自分→Chrome版→井上さん、と
順に投げ下ろすピラミッド式の禁止）。代わりに：

1. **毎回、最初に Agent ツールで検討チームを並行起動して全員会議を開く**
   （調査担当・代替設計担当など複数視点。例外なく毎タスク実施）。
2. 「誰が・どこを・どれだけ介入するのが最適か」を全員の結果を統合して決めてから動く。
3. 教訓例：Metaトークン問題は単独判断では「Secrets3つの手動登録」で停滞したが、
   並行検討で「無期限システムユーザートークンにより作業自体を消す」という上位解が出た。
   単独判断は局所最適に陥る。**作業を減らす最良の方法は、作業を不要にする設計を全員で探すこと。**

### 🔴 遊休ゼロの原則（2026-06-08 追加・ヒマを持て余すな）

- **担当タスクが人間待ち・外部待ちでブロックされたら、待っている間に余った稼働を
  運用ボードの他の優先タスクの補助に自動的に回す**（指示を待たない）。
- 例：トークン設定が井上さんの操作待ちの間に、次の優先タスク（商品一元管理のAPI調査、
  撤去計画の作成など）をAgentで並行起動して進めておく。
- セッション終了時、進めた補助作業も運用ボードに記録して次セッションへ引き継ぐ。

- 設定済みのAPIキー・Webhook・GAS連携などは、**人間を介さず直接呼び出す**。
- **コピペが必要な手順を提示する前に、まず自動化や直接連携で代替できないかを必ず検討し、
  代替できる場合はそちらを採用する。**
- これは秘書セッションに限らず、**各セッションが単独で動作している場合も常に適用する共通の前提**とする。
- 井上さんへの確認は、SNS投稿前の承認のような意図的な人間ゲートに限る。値の受け渡しのための
  手作業は原則ゼロを目指す。

### 🚫 井上さんに作業させてよい範囲（厳守・2026-06 の反省を反映）

井上さんに手を動かしてもらうのは、**技術的・権限的に本人にしかできない操作だけ**に限定する。
具体的には次のみ許容：
1. Google等の **OAuth/認証・アクセス許可ダイアログの承認**
2. **公開デプロイの最終確定ボタン**（内容確認のため本人が押す）
3. **Claude Code on the web の環境変数・環境設定の変更**（セッションからは変更不可）
4. SNS投稿前など、**意図的な人間ゲートとしての承認**

上記以外（値のコピペ、ファイルの受け渡し、台帳更新、アップロード等）は**すべてClaude Code側で自動実行する**。

**やってはいけないこと：**
- 同じコピペ手順を**何度も投げ直す**こと。1回で通らなければ即、別アプローチに切り替える。
- 委譲先（Chrome版等）に、**その委譲先が実際には実行できない作業**を振ること。
  振る前に「この相手はこの操作を物理的にできるか？」を必ず確認する。
- 作業を頼む前に必ず自問する：**「これは本当に井上さん本人にしかできないか？ 自動化できないか？」**
  少しでも自動化余地があれば、頼む前に自分で試す。
- **生の値（トークン・URL等）を井上さんに渡して「あなたが入力して」と丸投げすること。**
  値の入力・画面遷移は Chrome版／自動化に **「確定（保存）の直前まで」必ずやらせる**。
  井上さんに残してよいのは **最後の確定操作1つだけ**。
  （例：Claude Code環境変数の登録 → Chrome版が設定画面を開き名前と値を入力 →
   井上さんは保存ボタンを押すだけ。「不可だから本人がやれ」と全部投げない。
   本人に必要なのは"最終確定"であって"入力作業"ではない、を常に区別する。）
- **Chrome版プロンプトを作る前に必ず確認する：「井上さんが直接やる方が速くないか？」**
  Chrome版経由は「プロンプト作成 → 貼り付け → Chrome版の実行 → 報告」という4手順になる。
  「URLを開いて値を貼るだけ」なら井上さんに直接やってもらう方が圧倒的に速い。
  Chrome版が本当に必要なのは、ログインが要る画面操作・長い自動スクレイピング等に限定する。
